# AI 건강비서 앱 개발 가이드

## 1. 프로젝트 목표

AI 건강비서는 사용자가 대화, 사진, 처방전, 검사결과, PDF 또는 문서를 입력하면 내용을 분석해 건강기록으로 정리하는 Android 앱이다.

핵심 목표는 다음과 같다.

- 복잡한 입력 양식 없이 대화형으로 사용
- 처방전을 진료과별로 정리
- 복용약 중복 및 중복 처방 가능성 경고
- 약으로 유추되는 질환은 확정 질환과 분리 저장
- 건강상태 화면은 질환명 중심으로 간단하게 표시
- 사진, PDF, 문서, 음성 입력 지원
- 저장된 건강정보를 다음 분석에 활용
- 병원 일정과 복약 알림 관리

## 2. 기술 기준

| 항목 | 기준 |
|---|---|
| 플랫폼 | Android |
| 프레임워크 | Flutter |
| AI | Gemini REST API |
| 기본 모델 | `gemini-2.5-flash` |
| 로컬 저장 | SharedPreferences 및 보안 저장소 |

## 3. 주요 화면

### 3.1 AI 건강비서

- 자연어 대화
- 사진 촬영
- 갤러리 다중 선택
- PDF 및 문서 첨부
- 음성 입력
- AI 분석 결과 확인
- 중복 처방 경고

### 3.2 건강상태

건강상태는 지나치게 자세한 약 정보 대신 질환 중심으로 간단하게 보여준다.

표시 순서:

1. 확정 질환
2. 복용약으로 추정한 질환
3. 복용 중인 약 요약
4. 알레르기
5. 건강 메모

표시 예:

```text
확정 질환
- 당뇨
- 고혈압

복용약으로 추정한 질환
- 고지혈증
- 알레르기성 비염
- 말초혈관 또는 혈액순환 관련 질환

복용 중인 약
- 내과 7개
- 신경과 3개
- 안과 2개
```

다음과 같은 내부 데이터 구조는 화면에 직접 노출하지 않는다.

```text
{name: 당뇨, source: 글루코정, confidence: 0.8}
```

화면에는 `당뇨`처럼 사용자에게 필요한 이름만 표시한다. 근거와 상세 약 정보는 상세보기에서만 확인한다.

## 4. 처방전 관리

### 4.1 진료과별 분류

처방전은 다음과 같이 진료과별로 묶어 관리한다.

```text
내과
- 처방일
- 병원
- 의사
- 약 목록

신경과
- 처방일
- 병원
- 의사
- 약 목록

안과
- 처방일
- 병원
- 의사
- 약 목록
```

지원 진료과 예:

- 내과
- 신경과
- 안과
- 정형외과
- 피부과
- 이비인후과
- 비뇨의학과
- 정신건강의학과
- 기타
- 진료과 확인 필요

문서에 진료과가 없거나 확실하지 않으면 임의로 판단하지 않고 `진료과 확인 필요`로 저장한다.

### 4.2 처방전 저장 항목

처방전 한 건은 약 하나마다 따로 저장하지 않고 진료과와 처방일 기준으로 묶어서 저장한다.

- 진료과
- 병원명
- 담당 의사
- 처방일
- 약 목록
- 복용법 요약
- 관련 질환 추정
- 중복 처방 여부

약 상세정보는 내부적으로 저장하되 기본 화면에서는 간단히 표시한다.

내부 저장 가능 항목:

- 약 이름
- 성분
- 용량
- 복용 횟수
- 복용 시점
- 복용 기간
- 일반적인 용도

기본 화면 표시 예:

```text
내과 처방
2026-07-17 · 7개 약
당뇨, 혈압, 고지혈증 관련
```

상세보기를 누르면 약 이름과 복용법을 확인할 수 있도록 한다.

## 5. 질환 분류

질환은 반드시 다음 두 종류로 분리한다.

### 5.1 확정 질환

다음 경우에만 확정 질환으로 저장한다.

- 처방전 또는 진단서에 진단명이 명확히 표시됨
- 사용자가 직접 확정 질환이라고 입력함
- 병원 기록에 명확한 진단명이 있음

### 5.2 복용약으로 추정한 질환

약의 일반적인 용도에 따라 유추한 질환은 반드시 추정으로 저장한다.

예:

- 글루코정, 메트포르민 → 당뇨 가능성
- 혈압약 → 고혈압 가능성
- 스타틴 계열 → 고지혈증 가능성
- 항히스타민제 → 알레르기성 비염 가능성

표시 원칙:

- 화면에는 질환명만 간단히 표시
- 같은 질환은 하나로 통합
- 약 하나마다 같은 질환을 반복하지 않음
- 근거는 상세보기에서만 표시
- 추정 질환을 확정 진단처럼 표현하지 않음

## 6. 중복 처방 경고

새 처방전이 분석되면 기존 복용약과 비교한다.

검사 항목:

- 동일 약 이름
- 동일 성분
- 다른 상품명의 같은 성분
- 같은 계열 약 중복
- 처방 기간 중복
- 다른 진료과에서 동일 약 처방
- 기존 약을 대체한 새 처방 가능성

경고 단계:

### 주의

동일 성분 또는 같은 계열일 가능성이 있지만 정보가 부족한 경우

```text
중복 처방 가능성이 있습니다.
기존 복용약과 새 처방을 의사 또는 약사에게 확인하세요.
```

### 위험

동일 약 또는 동일 성분이 명확히 겹치는 경우

```text
중복 처방 가능성
같은 약 또는 같은 성분이 중복될 수 있습니다.
임의로 함께 복용하거나 중단하지 말고 의료진에게 확인하세요.
```

중복 여부는 진료과가 달라도 비교한다.

## 7. 중복 데이터 방지

중복 입력은 저장 전에 검사한다.

- 같은 처방전 사진
- 같은 PDF 또는 문서
- 같은 약과 용량
- 같은 병원 일정
- 같은 검사결과
- 같은 건강기록
- 같은 알림

처리 원칙:

- 완전 중복: 저장하지 않고 경고
- 일부 변경: 기존 기록 업데이트
- 같은 질환: 하나로 통합
- 같은 약: 진료과별로 묶되 전체 복용약에서는 중복 제거
- 공백, 대소문자, 기호 차이는 무시

기존에 잘못 저장된 객체 문자열도 앱 시작 시 자동 정리한다.

예:

```text
{name: 고혈압, source: 아젤라정}
```

정리 후:

```text
고혈압
```

## 8. 파일 및 입력 지원

지원 입력:

- 카메라 촬영
- 갤러리 사진
- 여러 사진 동시 선택
- PDF
- TXT
- CSV
- JPG
- JPEG
- PNG
- WEBP
- 음성 입력

여러 사진과 문서를 한 번에 첨부하면 하나의 처방 또는 건강 기록으로 종합 분석한다.

사진은 최대 5장을 기본으로 하며 동일 파일은 첨부 단계에서 제외한다.

## 9. AI 분석 규칙

Gemini는 다음 순서로 분석한다.

1. 문서 종류 판별
2. 병원과 진료과 확인
3. 처방일 확인
4. 약 목록 추출
5. 기존 복용약과 비교
6. 중복 처방 가능성 확인
7. 관련 질환 추정
8. 확정 질환과 추정 질환 분리
9. 진료과별 처방 기록 생성
10. 사용자에게 간단한 요약 제공

AI 응답 원칙:

- 내부 JSON을 사용자에게 그대로 표시하지 않음
- 자연스러운 한국어로 요약
- 너무 많은 약 정보를 한 화면에 나열하지 않음
- 질환명 중심으로 묶음
- 추정은 반드시 추정으로 표시
- 약 변경이나 중단을 직접 지시하지 않음

## 10. 건강상태 UI 원칙

현재 화면에서 가장 중요한 정보만 먼저 보여준다.

### 기본 카드

- 확정 질환
- 추정 질환
- 진료과별 복용약 수
- 알레르기
- 최근 주의사항

### 표시 제한

- 기본 화면에서 각 영역은 최대 5개까지만 표시
- 초과 항목은 `더보기`로 확인
- 약 목록은 진료과별 카드로 접기/펼치기
- 같은 질환은 한 번만 표시
- 약의 성분, 용량, 횟수는 상세보기에서 표시

## 12. Gemini 연결 안정화

네트워크 오류 대응:

- 최대 3회 자동 재시도
- `ClientException` 처리
- `SocketException` 처리
- 타임아웃 처리
- HTTP 429 및 500 이상 재시도
- 오류 로그에서 API 키 자동 숨김

사용자 메시지 예:

```text
AI 서버 연결이 일시적으로 중단되었습니다.
인터넷 연결을 확인한 뒤 다시 시도해 주세요.
```

## 13. Android 빌드 설정

`flutter_local_notifications` 사용을 위해 Android core library desugaring을 활성화한다.

Kotlin DSL 기준:

```kotlin
android {
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
        isCoreLibraryDesugaringEnabled = true
    }
}

dependencies {
    coreLibraryDesugaring("com.android.tools:desugar_jdk_libs:2.1.5")
}
```

GitHub Actions 빌드 순서:

1. Flutter Android 프로젝트 생성
2. 앱 소스 복원
3. Android 권한 및 앱 이름 적용
4. desugaring 설정 적용
5. `flutter pub get`
6. `flutter analyze`
7. `flutter build apk --release`
8. APK artifact 업로드

## 14. 데이터 안전 원칙

- 앱은 의료진의 진단과 치료를 대체하지 않음
- 추정 질환은 확정 진단으로 표시하지 않음
- 약 중단, 증량, 감량을 임의로 지시하지 않음
- 중복 처방 발견 시 의사 또는 약사 확인을 안내
- API 키와 건강정보를 오류 로그에 그대로 남기지 않음
- 사용자가 저장 여부를 확인할 수 있도록 함
- 삭제 및 수정 기능 제공

```text
건강상태
```
